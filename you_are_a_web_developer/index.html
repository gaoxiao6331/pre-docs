重新认识浏览器：DOM 模式​
浏览器本质上是一个高性价比的绘图程序，DOM 模式是浏览器给定的一套标准与规则。​
主要的两种开发方式：​
1. 写 HTML 标签，告诉浏览器帮我渲染一个矩形（声明式）。​
2. 用 JS 调用浏览器 API 创建节点，并插入到渲染树里（命令式）。​
劣势：早期设计导致的性能瓶颈，上限较低，深度优化困难。​
// 1. 声明式 (HTML)​
<div id="app">​
<h1>Hello, DOM</h1>​
</div>​
// 2. 命令式 (JS API)​
const btn = document.createElement('button');​
btn.textContent = 'Click Me';​
document.getElementById('app').appendChild(btn);​

重新认识浏览器：Canvas 模式​
追求极致性能的“画布”模式。​
浏览器只提供基础能力，所有元素的组装与渲染均由你自行定义。​
想要极致性能就必须走这条路，但开发成本极高，且与原生应用相比仍有差距。​
浏览器实现基础绘图 API，开发成本稍低，但不方便进行 3D 场景渲染。​
2D​
桥接 OpenGL 渲染库进行绘图。操作较为麻烦，在 WebGPU 普及后将逐步被废弃。​
WebGL​
为了解决 WebGL 性能等问题，提供直接调用 GPU 画图的能力，绝对的未来主流。​

框架的本质：Vue 与 React​
现代框架说白了就是协助操作 DOM 的工具。​
开发者只需定义好状态（数据）和渲染函数，数据一旦发生改变，​
框架的核心引擎便会自动更新界面。从此告别繁琐的手动 DOM 操作，实现“数据驱动视图”。​

架构进阶：微前端 (Micro-Frontend)​
大型团队分治与整合的架构利器。​
解决痛点：项目庞大时，代码耦合导致维护困难，部署牵一发而动全身。​
分离与整合：允许不同团队分开开发独立子应用，最后像拼图一样在主应用中整合。​
沙箱隔离：在复杂场景下实现样式（CSS）与全局变量（JS）的彻底隔离，避免互相冲突搞崩整个页面。​

架构进阶：BFF 与 SSR​
​
BFF (Backend For Frontend)​
专为前端服务而设立的后端。​
接口裁剪与聚合：一个接口返回首屏所需的所有数据，减小传输体积，避免发起多次请求。​
逻辑转移：客户端的数据加工移至内网服务端执行，CPU 算力强且网络延迟极低。​
​
SSR (服务端渲染)​
提前在服务器完成首屏 HTML 拼装。​
工作机制：代码先在服务器跑一遍，该执行的逻辑和获取的数据全拿完，然后直接下发生成好的 HTML 内容。​
核心收益：浏览器只需解析 HTML，速度极大提升，显著改善白屏时间与搜索引擎优化(SEO)。​
​
进阶必知：事件循环 (Event Loop)​
​
理解 JavaScript 的异步执行机制 是进阶前端的分水岭。​
运行基础：JS 是单线程的，浏览器通过任务队列调度执行。​
宏任务 (Macrotask)：如整体脚本、setTimeout、事件交互等。每次执行一个宏任务。​
微任务 (Microtask)：如 Promise.then。执行完一个宏任务后，必须优先清空所有的微任务队列，再进入下一个宏任务的执行或者渲染。​
​
进阶必知：浏览器渲染流水线​
理解从 代码到屏幕像素 的完整链路对于排查卡顿至关重要。​
核心流程：HTML 生成 DOM，CSS 生成 CSSOM，合并为 Render Tree（渲染树），经过 Layout（布局计算排版）最后被 Paint（绘制）成图层。​
性能洞察：JS 执行可能阻塞渲染；流畅动画请使用 requestAnimationFrame；职业选手会关注 LCP 与 CLS 指标。​
​
​
性能优化与排障策略​
核心排障法：使用浏览器开发工具，录制并分析 Performance 火焰图 查找瓶颈​
​
网络通信优化​
1. 不请求： 充分利用浏览器缓存。​
2. 少请求： 聚合接口(BFF)、压缩资源体积。​
3. 提前请求： DNS 预解析、TCP 预连接、核心业务数据资源预拉取 (Prefetch)。​
​
代码计算优化​
1. 不计算： WebWorker 在后台线程跑，或交给后端算。​
2. 少计算： 任务拆分，避免阻塞主线程的事件循环。​
3. 提前计算： 利用缓存与预计算，在用户触发前完成核心结果准备。​
4. 算快点： 用 Wasm 或 WebGPU 算。​
